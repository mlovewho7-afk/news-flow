# FinancialJuice 뉴스 플로우 — 설계

**날짜:** 2026-07-16
**상태:** 설계 승인 대기

## 배경 (Why)

사용자가 https://www.financialjuice.com 스타일의 실시간 금융 뉴스 티커를 본인 홈페이지에 갖고
싶어함. 10분 간격 자동 갱신을 우선 샘플로 붙이고, 이후 다른 뉴스 소스로 확장할 계획.

## 조사 결과 (설계에 영향을 준 사실)

- FinancialJuice의 실시간 스트림(`wss://rt.financialjuice.com`)은 로그인한 사용자의
  `feedtoken`으로만 인증되는 것으로 보이는 WebSocket. 비로그인 상태에선 `feedtoken`이 빈
  문자열이었고 뉴스 조회용 PageMethod도 없어, **비로그인 자동화로 접근이 사실상 어려운 것으로
  판단됨**(실제 WebSocket 연결을 시도해 거부를 확인한 것은 아니고, 페이지 소스 정황상의 결론).
- 반면 `https://www.financialjuice.com/feed.ashx`는 **로그인 없이 공개된 RSS 피드**임을
  확인함 (200 응답, `text/xml`, robots.txt 파일 자체가 존재하지 않아 차단 명시가 없음 — 단,
  "허용됨"을 뜻하진 않음). 최신 뉴스가 실시간으로 반영됨(테스트 시점 기준 발행 40초 전
  기사까지 포함).
- 이 엔드포인트는 Cloudflare 뒤에 있음. 실측 중 짧은 시간에 5회 연속 요청했을 때
  `feedid=3/4/5` 요청에서 Cloudflare error 1015(rate-limit)가 발생함을 확인. 다만 이게
  `feedid` 파라미터 자체 때문인지 단순 요청 빈도 때문인지는 분리 확인하지 못함. **10분 간격은
  이보다 훨씬 느슨해 안전할 것으로 예상되나, 장시간(수 시간~수일) 지속 시의 실측은 아직 없음**
  — 초기 운영 중 1015/429가 재발하는지 모니터링 필요.
- 피드에는 카테고리 필드가 없음 (로그인 후 웹사이트에만 Macro/Bonds/FX 등 탭 필터가 있고,
  공개 RSS는 통합 피드 하나뿐). `feedid` 쿼리 파라미터로 필터링을 시도했으나 `feedid=1`과
  `feedid=2`가 동일한 전체 피드를 반환해, 최소한 이 값들로는 필터링되지 않음을 확인.
- **제3자 상용 뉴스 사이트 재게시의 리스크**: robots.txt 미차단은 크롤러 예절 관례일 뿐
  이용약관·저작권 문제와는 별개다. 이 설계는 그 리스크를 기술적으로 "허용됐다"고 해소한 게
  아니라, 사용자가 이미 인지한 상태에서 개인용 샘플로 진행하기로 한 것으로 취급한다(아래
  "재게시 리스크" 절 참고).

## 아키텍처

새 독립 저장소 (가칭 `news-flow` — **이 이름은 메인 에이전트가 브레인스토밍 중 임시로 붙인
것이며 사용자가 확정하지 않았다. 구현 착수 전 사용자에게 실제 저장소명을 확인해야 한다.**),
GitHub Pages, 기존 `macro-dashboard`와 별개 저장소·별개 URL.

결과물의 흐름("데이터 갱신 → git commit/push → GitHub Pages 재배포")은 기존 매크로 대시보드
프로젝트의 방식과 모양은 비슷하다(그 프로젝트는 `data-viz-project` 저장소에서 로컬 Python
스크립트를 실행해 `macro-dashboard` 저장소로 git push하는 2단계 구조). 하지만 **트리거
방식은 다르다** — 기존 방식은 사용자가 로컬에서 수동으로(또는 명시적 요청 시) 실행하는
반면, 이 설계는 GitHub Actions 크론으로 10분마다 자동 실행되는 방식을 새로 도입한다. 이는
이 프로젝트에서 처음 쓰는 검증되지 않은 자동화 방식이므로, 아래 "검증 계획"에서 수동
트리거로 먼저 확인한 뒤 크론을 켠다.

```
news-flow/
├── .github/workflows/fetch-news.yml   ← 10분 크론 (concurrency 그룹 지정, 겹침 실행 방지)
├── scripts/
│   ├── sources/financialjuice.py      ← 소스별 fetch+parse (다음 소스 추가 시 이 옆에 파일만 추가)
│   └── fetch_news.py                  ← sources/*를 호출해 병합·중복제거·저장
├── data/news.json                     ← 누적 데이터 (Non-destructive, 상한 없음 — 아래 참고)
├── index.html                         ← 정적 페이지 (티커 UI + 열람용 가림막)
└── docs/superpowers/specs/            ← 이 설계 문서
```

### 파이프라인 (GitHub Actions가 10분마다 실행)

1. `https://www.financialjuice.com/feed.ashx` GET (User-Agent 지정)
2. RSS XML 파싱 → 각 `<item>`에서 `guid`, `title`, `link`, `pubDate` 추출
3. `data/news.json`을 읽어 기존 `guid` 집합과 대조, 신규 항목만 추가
4. `pubDate` 기준 내림차순 정렬. **상한(캡)은 지금 두지 않는다** — 사용자가 "우선 샘플로
   테스트"한다고 밝힌 단일 소스 단계에서, 파일 크기나 로딩 속도가 실제 문제가 됐다는 근거가
   아직 없다(MEASURE-FIRST). 삭제 정책은 실측 근거(파일 크기·로딩 시간)가 생겼을 때 별도로
   설계해 사용자 확인을 받고 도입한다(REVERSIBLE — 지금은 되돌릴 수 없는 삭제 정책을 만들지
   않는 쪽을 택함). 이 "당분간 무상한" 정책은 고정이 아니라, 다른 뉴스 소스를 추가하는 시점
   등 스코프가 커질 때 재검토 대상이다(위 "범위 밖" 참고)
5. 새 항목이 하나도 없으면 **커밋을 건너뛰고 정상 종료**(빈 diff에 `git commit`을 강행하면
   실패 종료하므로, 뉴스가 뜸한 시간대마다 워크플로우가 "실패"로 표시되는 소음이 생김 — 이를
   막기 위해 워크플로우가 `git diff --quiet` 등으로 변경 여부를 먼저 확인). 새 항목이 있을
   때만 `data/news.json` 커밋 + push → Pages 자동 재배포

**실패 처리 (FAIL-LOUD, 각 단계별):**
- `feed.ashx` 요청 실패(네트워크 오류·429·5xx) 또는 RSS 파싱 실패(예: Cloudflare가 challenge
  페이지를 200으로 반환하는 경우) → 워크플로우는 실패로 종료하고 기존 `data/news.json`은
  건드리지 않음. 조용히 빈 파일로 덮어쓰지 않음
- git push 충돌(동시 실행) → `.github/workflows/fetch-news.yml`에 `concurrency:` 그룹을
  지정해 이전 실행이 끝나기 전 다음 스케줄 실행이 겹치지 않도록 함(애초에 충돌 상황 자체를
  방지)
- 신규 항목 없음은 실패가 아니라 정상 상태 (위 5단계 참고)

### 카테고리 태깅 (best-effort, 오분류 감수)

공식 카테고리 필드가 없으므로 제목 텍스트 키워드 매칭으로 추정 분류한다. **이 방식은 필연적으로
오분류를 만든다** — 예를 들어 "strike"는 군사 공격뿐 아니라 노동 파업 기사에도, "war"는
전쟁뿐 아니라 무역전쟁(trade war) 같은 일상적 매크로 기사에도 걸려 평범한 뉴스가 Breaking으로
잘못 강조될 수 있다. 단일 소스로 "우선 샘플 테스트"하는 이번 단계에서는 이 오분류율을 감수하고
진행하되, 실제 운영해보고 눈에 띄게 부정확하면 키워드를 조정하거나(코드 수정만 필요, 재배포는
git push로 즉시 반영) 필요하면 카테고리 자체를 줄이는 것으로 대응한다.

| 태그 | 키워드 예시 |
|---|---|
| Breaking | strike, explosion, sirens, military, attack, war |
| Macro | Fed, CPI, PPI, GDP, central bank, rate, inflation |
| Bonds | yield, treasury, bond, curve |
| FX | USD, EUR, JPY, yuan, forex, currency pair 패턴 |
| Corporate | 기업명 티커($ 표기), earnings, M&A |
| Other | 위 어디에도 안 걸리면 기본값 |

키워드 목록은 `scripts/sources/financialjuice.py` 상단에 상수로 SSOT — 태깅 로직 수정 시
여기만 고침.

### 프론트엔드 (`index.html`)

- 브레인스토밍에서 승인된 A안(다크 실시간 티커) 그대로 구현
- 카테고리 탭으로 필터링 (전체/Breaking/Macro/Bonds/FX/Corporate/Other — 위 카테고리 태깅
  절 참고, 사용자 확정이 아니라 메인 에이전트 제안값)
- `pubDate`(GMT)를 KST(UTC+9)로 변환해 표시 (사용자가 한국 기준으로 쓸 것으로 가정한
  메인 에이전트 제안값 — 명시적으로 확정받은 사항은 아님)
- 클라이언트가 `data/news.json`을 fetch, 페이지가 열려 있는 동안 60초마다 재요청해 화면 갱신
  (서버 파이프라인은 10분 주기지만, 페이지는 그보다 자주 확인해 새 데이터가 배포되는 즉시
  반영되게 함)
- Breaking 항목은 좌측 빨간 강조 + 배경색 처리(목업 A안과 동일)

### 열람 가림막 (접근 제어 아님 — 표현 정정)

비밀번호는 사용자가 확정한 대로 기존과 동일한 `kiwoom77`을 재사용한다. 구현 방식(SHA-256
해시 검증 + `localStorage` 플래그로 최초 1회만 입력받는 방식)은 macro-dashboard가 실제로
이렇게 구현돼 있다는 전제로 메인 에이전트가 제안한 것이며, 착수 시 macro-dashboard의 실제
코드를 확인해 다르면 여기 맞춰 조정한다. 어느 쪽이든 평문 비밀번호는 코드에 남기지 않는다.

**이건 진짜 접근 제어가 아니라 "우연히 URL을 아는 사람이 바로 못 보게 하는" 수준의 열람
가림막임을 명확히 해둔다.** 해시값이 페이지 소스에 그대로 노출돼 있어 우회 가능하고, 무엇보다
이 가림막은 `index.html`(화면)만 가릴 뿐 `data/news.json`은 정적 파일로 URL만 알면 누구나(다른
크롤러 포함) 가림막 없이 직접 가져갈 수 있다. 실질적 비공개가 필요하다면 저장소 자체를
비공개로 두는 방법을 검토해야 하지만, GitHub Pages 무료 플랜은 통상 공개 저장소를 요구하므로
이번 설계에서는 다루지 않는다(범위 밖).

### 재게시 리스크 (있는 그대로 인정)

FinancialJuice는 사용자가 소유·계약한 서비스가 아닌 제3자 상용 뉴스 사이트다. robots.txt
미차단은 크롤러 예절 관례일 뿐이며 이용약관·저작권 문제를 해소해주지 않는다. 저장소가 공개
저장소로 운영될 가능성이 높다는 점을 고려하면, 재게시된 콘텐츠와 10분마다의 커밋 이력이 공개
git 히스토리에 남는다 — 나중에 데이터를 지우거나 저장소를 내려도 과거 커밋은 히스토리
재작성 없이는 지워지지 않는다. 이 설계는 이 리스크를 기술적으로 해소된 것으로 포장하지
않는다. 사용자가 "우선 개인용 샘플로 테스트"하는 것으로 이미 리스크를 인지한 상태에서
진행하기로 한 결정이며, 앞의 "열람 가림막" 절에서 설명했듯 그 가림막이 실질적 비공개를
보장하지 않는다는 점도 함께 인지해야 한다.

## 범위 밖 (Out of scope, 이번 설계에서 제외)

- 다른 뉴스 소스로의 확장 — 사용자가 "우선 이것부터 테스트하고 확장하겠다"고 명시. 위
  아키텍처의 `scripts/sources/` 디렉터리 구조가 소스별 파일을 나란히 추가할 수 있는 최소한의
  확장 지점이지만, 여러 소스를 병합·우선순위 처리하는 로직 자체는 지금 만들지 않음(YAGNI —
  실제로 두 번째 소스가 필요해질 때 설계)
- 뉴스 검색·북마크·알림 등 FinancialJuice의 로그인 전용 기능
- `macro-dashboard`(CPI/GDP/Taylor) 파이프라인과의 통합 — 완전히 별도 저장소·별도 URL로
  운영하여 기존 파이프라인이 이 저장소를 건드릴 일이 없고, 반대도 마찬가지
- 저장소를 비공개로 전환하는 것(GitHub Pages 유료 플랜 필요) — 위 "재게시 리스크" 참고

## 검증 계획

- `scripts/fetch_news.py`를 로컬에서 1회 실행해 `data/news.json`이 정상 생성되고 `guid` 중복
  없이 최신순 정렬되는지 확인
- 같은 스크립트를 새 항목 없이 2회 연속 실행해, 두 번째 실행이 파일을 변경하지 않고 정상
  종료하는지 확인 (커밋 스킵 로직이 실제로 빈 diff를 만드는지)
- `feed.ashx`를 일부러 접근 불가능한 URL로 바꿔 실행해, 기존 `data/news.json`이 훼손되지 않고
  스크립트가 실패로 종료하는지 확인 (요청 실패 경로의 FAIL-LOUD 검증)
- 응답 본문을 유효하지 않은 XML(예: HTML challenge 페이지)로 바꿔치기해 실행해, 파싱 실패
  시에도 기존 `data/news.json`이 훼손되지 않고 실패로 종료하는지 확인 (파싱 실패 경로의
  FAIL-LOUD 검증 — 200 응답이라 겉보기엔 성공처럼 보이므로 별도로 확인)
- 몇 가지 실제 기사 제목으로 카테고리 태깅 함수를 단위 테스트해, 표에 정의한 키워드가 의도한
  태그로 매핑되는지 확인 (오분류 자체를 없애는 게 아니라, 최소한 매핑 로직 자체엔 버그가
  없는지 확인하는 수준)
- GitHub Actions 워크플로우를 수동 트리거(`workflow_dispatch`)로 1회 실행해 커밋·push가
  정상 동작하는지 확인 후 크론 스케줄 활성화
- `index.html`을 브라우저로 열어 열람 가림막·티커 렌더링·카테고리 필터·KST 시간 변환이
  실제로 동작하는지 확인
